# Расширенный аудит пайплайна Jira: Проблемы слоев Raw и Clean

В данном документе описаны критические проблемы, выявленные в процессе анализа архитектуры и реализации слоев данных (Bronze/Silver) для интеграции с Jira. Список расширен на основе анализа пустующих таблиц и отсутствующих сущностей.

---

## Слой RAW (Bronze)

### 1. Риск рассинхронизации инкрементальной загрузки
*   **Проблема**: Метод `_safe_updated_jql_value` использует фиксированное окно ретроспективы.
*   **Причина**: Использование `updated >= "{timestamp}"` не учитывает удаленные задачи.
*   **Последствия**: В БД остаются "призраки" (удаленные в Jira задачи).
*   **Фикс**: Внедрить периодическую полную сверку ID (Full Sync).

### 2. Отсутствие обработки лимитов API (Rate Limiting)
*   **Проблема**: Прямой запрос без логики повторов.
*   **Последствия**: Падение пайплайна при ошибке 429.
*   **Фикс**: Использовать `tenacity` или встроенные ретраи `dlt`.

### 3. Высокое потребление памяти
*   **Проблема**: Запрос всех полей и чейнджлога в одном потоке.
*   **Последствия**: Риск OOM на больших объемах.
*   **Фикс**: Ограничить `fields` и вынести чейнджлог в отдельный ресурс.

## Слой CLEAN (Silver) — Пустующие таблицы (Есть в схеме, но нет данных)

### 4. Отсутствие синхронизации комментариев (`comments`, `comment_issues`)
*   **Проблема**: Таблицы созданы, но ассет для их наполнения в `clean.py` отсутствует.
*   **Причина**: Логика парсинга вложенного объекта `comment` из сырых данных не реализована.
*   **Последствия**: Невозможность анализировать обсуждения и извлекать блокировки из текста.
*   **Фикс**: Создать ассет `clean_jira_comments`, нормализующий данные из `raw_jira.issues__fields__comment__comments`.

### 5. Мертвые связи между задачами (`relation_issue_issues`, `relation_issue_types`)
*   **Проблема**: Данные о линках (blocks, relates to) не переносятся.
*   **Причина**: Пайплайн игнорирует поле `issuelinks`.
*   **Последствия**: Невозможно построить граф зависимостей и выявить критический путь.
*   **Фикс**: Реализовать ассет для парсинга связей и типов линков.

### 6. Отсутствие истории релизов (`release_changelog`)
*   **Проблема**: Таблица истории изменений версий пуста.
*   **Последствия**: Нельзя отследить, когда и почему менялись даты релиза.
*   **Фикс**: Добавить логику отслеживания изменений в ассет версий.

### 7. Пустые роли пользователей в задачах (`jira_user_issue_roles`)
*   **Проблема**: Нет данных о том, кто является Assignee, Reporter или Creator в терминах UUID системы.
*   **Причина**: Код не маппит внешние ID пользователей из полей задачи на UUID в `jira_users`.
*   **Последствия**: Невозможно фильтровать метрики по конкретным исполнителям.
*   **Фикс**: Настроить маппинг `assignee_id`, `reporter_id` и `creator_id`.

## Проблемы схемы (Отсутствующие сущности)

### 8. Отсутствие сущности "Приоритет" (`priorities`)
*   **Проблема**: В схеме нет таблицы для приоритетов.
*   **Причина**: Приоритет сейчас либо игнорируется, либо хранится как сырой текст.
*   **Последствия**: Невозможность группировки и взвешенного анализа (например, SLA по критическим задачам).
*   **Фикс**: Создать таблицу `clean_jira.priorities` и добавить `priority_id` в `issues`.

### 9. Отсутствие поддержки лейблов (`labels`)
*   **Проблема**: Нет таблицы для тегов (labels).
*   **Причина**: Поле `labels` является массивом строк, что требует нормализации в M2M связь.
*   **Последствия**: Потеря возможности гибкой фильтрации задач по произвольным признакам.
*   **Фикс**: Создать таблицу `labels` и связующую таблицу `issue_labels`.

### 10. Слабая модель иерархии (Родители/Дети)
*   **Проблема**: Иерархия ограничена полем `parent_id` в таблице `issues`.
*   **Причина**: Простая ссылка на родителя не поддерживает глубокую вложенность (Epic -> Capability -> Feature -> Story).
*   **Последствия**: Трудности с агрегацией метрик (например, расчет Lead Time для Эпика как суммы вложенных элементов).
*   **Фикс**: Внедрить таблицу путей (Closure Table) или рекурсивные связи для иерархии.

### 11. Отсутствие единой таблицы блокировок
*   **Проблема**: Блокировки размазаны между `issuelinks` (системные) и `comment_blockings` (из комментариев).
*   **Последствия**: Сложность расчета "времени простоя" задачи из-за внешних факторов.
*   **Фикс**: Создать унифицированную таблицу `issue_blockings`, которая собирает данные из всех источников.

## Архитектура и Логика

### 12. Хардкод UUID родительского проекта
*   **Проблема**: Все Jira-проекты привязаны к одному ID `000...001`.
*   **Фикс**: Динамический маппинг через `platform_project_id`.

### 13. Хрупкий маппинг уровней иерархии
*   **Проблема**: Определение Epic/Story через `ILIKE '%epic%'`.
*   **Фикс**: Использовать `hierarchyLevel` из метаданных Jira.

### 14. Отсутствие Worklogs
*   **Проблема**: В схеме нет таблиц для учета фактического времени.
*   **Фикс**: Добавить `clean_jira.worklogs`.

### 15. Потеря истории изменений спринтов
*   **Проблема**: `sprint_changelog` фиксирует только закрытие.
*   **Фикс**: Внедрить SCD Type 2 для спринтов.

### 16. Магическое число ID поля спринта (`customfield_10020`)
*   **Фикс**: Автоматический поиск ID поля по типу в метаданных.

### 17. Дублирование пользователей
*   **Проблема**: Один человек создается отдельно для каждого проекта.
*   **Фикс**: Сделать `jira_users` глобальной таблицей.

### 18. "Слепота" к удаленным статусам
*   **Проблема**: Пропуск чейнджлога при отсутствии статуса в справочнике.
*   **Фикс**: Добавить технический статус `Unknown`.

### 19. Чрезмерная очистка чейнджлога спринт-задач
*   **Проблема**: `TRUNCATE` при каждом запуске.
*   **Фикс**: Использовать инкрементальный `UPSERT`.

### 20. Фильтрация сложных кастомных полей
*   **Проблема**: Игнорирование глубоких вложенностей DLT.
*   **Фикс**: Использовать белый список полей вместо фильтрации по глубине.

### 21. Типизация внешних ID как Text
*   **Последствия**: Низкая производительность индексов.
*   **Фикс**: Использовать числовые типы или специализированные индексы.

### 22. Отсутствие контроля целостности (Checksums)
*   **Проблема**: Нет сверки `count(*)` между слоями.

### 23. Отсутствие метаданных о синхронизации
*   **Проблема**: Нет отслеживания `watermarks` для инкрементальной загрузки.

### 24. Сложность сопровождения SQL-запросов
*   **Проблема**: Огромные SQL-строки в Python-коде.
*   **Фикс**: Перенос трансформаций в `dbt`.

### 25. Отсутствие поддержки кросс-проектных связей
*   **Проблема**: Жесткая фильтрация по ключу проекта.

### 26. Неконсистентность поля `is_active` в спринтах
*   **Проблема**: Не учитывается статус задачи при закрытии спринта для Velocity.

### 27. Отсутствие нормализации "Resolution"
*   **Проблема**: Поле Resolution (Fixed, Won't Fix) не вынесено в справочник.
*   **Последствия**: Невозможность качественного анализа дефектов и причин закрытия задач.
*   **Фикс**: Добавить таблицу `resolutions` и связь в `issues`.
