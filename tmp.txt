  TD-002: Стандартизация транзакций

  Проблема — два несовместимых паттерна в clean layer:

  # Паттерн A (правильный — атомарный)
  with engine.begin() as conn:
      conn.execute(...)  # auto-commit при выходе, rollback при исключении

  # Паттерн B (текущий — ручной commit)
  with engine.connect() as conn:
      conn.execute(...)
      conn.commit()      # если упадёт до этого — данные не запишутся, ОК
                         # но если несколько conn.commit() — возможна частичная запись


  В supplementary.py строка 447 — conn.commit() внутри условного блока внутри одного engine.connect(). Если
  sprint_field_id есть — коммит происходит в середине asset'а, до второй части работы. Если потом упадёт — половина
  данных уже записана.

  issues.py использует несколько отдельных with engine.connect() блоков — каждый независимый commit, аналогичная
  ситуация.

  Риск: не катастрофический сейчас (большинство операций идемпотентны через ON CONFLICT), но создаёт неочевидное
  состояние при ошибках.

Суть проблемы (TD-001): В Clean-слое сейчас настроено полное сканирование (full re-scan) сырых данных Jira при каждом запуске. По мере роста количества задач (свыше 50 тыс.) ежечасные обновления начнут сильно тормозить или падать. Что нужно сделать:

Реализовать инкрементальное обновление с использованием high-watermark (сохранение состояния последней загрузки).
Использовать поля _dlt_load_id или fields__updated в качестве фильтров для выборки только новых или изменившихся записей с момента последнего запуска.
3. Надежность и целостность данных
Устранение состояний гонки (TD-003): Сейчас джоб обновления метрик работает независимо от джоба обновления Clean-слоя. Возможна ситуация, когда метрики начнут рассчитываться до того, как обновятся все очищенные данные, что приведет к неконсистентным отчетам.

Решение: Добавить в Dagster сенсор (sensor), который будет задерживать запуск обновления метрик до успешного завершения пайплайна Clean-слоя.
Атомарность транзакций (TD-002): В кодовой базе используются два разных подхода к транзакциям (например, в issues.py — один коммит в конце, а в supplementary.py — несколько).

Решение: Привести архитектуру к единому стандарту: одна транзакция на ассет через engine.begin() (с автокоммитом при успехе). Нужно избавиться от ручных вызовов conn.commit().
4. Оптимизация нагрузки на источники (TD-004)
Сейчас процесс загрузки из Jira вытягивает тяжелые payload'ы и использует длинные окна перезагрузки по умолчанию.

Если есть договоренность с продуктовой стороной, стоит разделить стратегию экстракции:
Легкие и быстрые ежедневные (daily) запуски с коротким временным окном.
Периодические "тяжелые" бэкфиллы (backfill) с широким временным окном для глубокой синхронизации данных.

✦ Если смотреть за пределы уже упомянутого технического долга, я бы порекомендовал обратить внимание на следующие
  аспекты, которые сделают платформу более зрелой и удобной для эксплуатации:

  1. DORA Metrics (Завершение реализации)
  В README.md указано, что метрики DORA (Deployment Frequency, Lead Time for Changes) находятся в статусе "in progress".
   * Рекомендация: Интегрировать данные из GitLab (через dlt) для отслеживания пайплайнов и релизов. Это позволит
     считать честный Lead Time for Changes (от коммита до продакшена), а не только Cycle Time задачи в Jira.

  2. Оптимизация обработки данных (Polars-first)
  В pyproject.toml уже есть polars, но во многих местах всё ещё может использоваться pandas или чистый SQL.
   * Рекомендация: Полностью перевести тяжелые вычисления в pipelines/calculations/ на Polars. Это даст кратный прирост
     скорости на больших объемах данных и снизит нагрузку на память.

  3. Расширение Admin API и UI
  Сейчас проект выглядит как мощный бэкенд с Metabase сверху, но управление ими может быть разрозненным.
   * Рекомендация:
       * Добавить в FastAPI эндпоинты для управления расписаниями Dagster (через dagster-graphql или клиент).
       * Реализовать "Health Dashboard" прямо в админке, который будет показывать статус свежести данных (Data
         Freshness) — когда в последний раз успешно обновлялась каждая метрика.

  4. Улучшение Observability (Наблюдаемость)
   * Рекомендация: Внедрить структурированное логирование (например, через structlog) и интегрировать его с Dagster.
     Сейчас при падении ассета может быть сложно быстро понять, на каком именно ID задачи в Jira произошла ошибка, без
     глубокого копания в логах контейнера.

  5. Безопасность и Multi-tenancy
  Поскольку проект позиционируется как self-hosted для организаций:
   * Рекомендация: Реализовать поддержку нескольких организаций или департаментов (Row Level Security в Postgres). Это
     позволит одной инсталляции платформы обслуживать разные команды, гарантируя, что менеджеры одной команды не видят
     детальные данные другой (если это требуется политикой компании).

  6. Документация метрик ("Data Dictionary")
  Для пользователей (менеджеров) часто важно понимать, как именно считается "Cycle Time" или "Velocity" (учитываются ли
  выходные, что считается за "Done").
   * Рекомендация: Создать в интерфейсе или в отдельном docs/metrics.md живой справочник определений, который
     автоматически обновляется на основе метаданных из кода расчетов.

Да, есть ещё несколько сильных направлений сверх этих 6:

  1. Ввести SLO/SLI для пайплайна.
     Минимум: freshness, % успешных раннов, длительность sync по проектам.
  2. Нормальный observability-стек.
     Структурные логи + метрики + алерты (хотя бы на падение jira_raw/jira_clean и stale-данные).
  3. Контракты данных на уровне схем.
     Проверки “обязательные поля/типы/уникальность” как обязательный gate перед расчётом метрик.
  4. Резервное копирование и DR-процедура.
     Автобэкапы Postgres + документированный restore drill (не только “backup есть”, а проверенный restore).
  5. Версионирование BI-пака и миграций как release-процесс.
     Явный changelog: “какие метрики/срезы изменились и с какого релиза”.
  6. Нагрузочный профиль.
     Регулярный synthetic-benchmark на 10k/50k/200k issues, чтобы видеть деградацию до прод-инцидента.
  7. Безопасность секретов.
     Уйти от plaintext/env где возможно: централизованный secret provider + ротация токенов + аудит.
  8. Дорожная карта multi-source заранее (TD-005).
     Даже до полной реализации — зафиксировать интерфейс universal.* и правила маппинга, чтобы не множить Jira-specific
     логику.
