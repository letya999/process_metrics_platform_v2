# Skill: Raw Layer (dlt Ingestion)

The raw layer is the Bronze tier — data lands here untransformed from Jira.
See also installed marketplace skill `dlt-skill` and `dlt-dagster` for general dlt patterns.

---

## What Lives Here

`raw_jira` schema — tables are auto-generated by dlt. Never create them manually.

Key tables created by dlt:
| Table | Content |
|---|---|
| `raw_jira.issues` | Flattened issue fields |
| `raw_jira.issues__fields__customfield_10020` | Sprint field (nested) |
| `raw_jira.changelogs` | Issue change history |
| `raw_jira.changelogs__items` | Individual change items |
| `raw_jira.sprints` | Sprint metadata |
| `raw_jira.boards` | Board definitions |

dlt adds metadata columns to every table:
- `_dlt_id` — row-level dedup key
- `_dlt_load_id` — batch identifier (links to `_dlt_loads` table)
- `_dlt_root_id` — for nested tables, links to parent row

## Dagster Asset: `raw_jira_data`

Location: `pipelines/assets/jira/raw.py`

This asset wraps a dlt pipeline run. Structure:

```python
@asset(
    group_name="jira_raw",
    compute_kind="python",
    description="Ingest raw Jira data via dlt into raw_jira schema",
)
def raw_jira_data(database: DatabaseResource) -> None:
    # dlt uses its own PostgreSQL destination configured via env vars:
    # DESTINATION__POSTGRES__CREDENTIALS__*
    pipeline = dlt.pipeline(
        pipeline_name="jira_raw",
        destination="postgresql",
        dataset_name="raw_jira",
    )
    ...
```

## dlt Destination Configuration

dlt reads PostgreSQL credentials from environment, NOT from `DATABASE_URL`. They are separate:

```env
# dlt destination credentials (raw_jira schema)
DESTINATION__POSTGRES__CREDENTIALS__HOST=postgres
DESTINATION__POSTGRES__CREDENTIALS__PORT=5432
DESTINATION__POSTGRES__CREDENTIALS__DATABASE=process_metrics
DESTINATION__POSTGRES__CREDENTIALS__USERNAME=postgres
DESTINATION__POSTGRES__CREDENTIALS__PASSWORD=postgres
```

These must be set even if `DATABASE_URL` is set. They are distinct config paths.

## Adding a New Jira Endpoint to Raw Ingestion

1. Define a dlt resource in `raw.py`:

```python
@dlt.resource(
    name="my_new_endpoint",
    write_disposition="replace",   # or "append" for changelogs
    primary_key="id",
)
def jira_my_new_endpoint(jira_client: JiraClient):
    """Fetch my new data from Jira."""
    page = 0
    while True:
        results = jira_client.get_my_endpoint(startAt=page * 50, maxResults=50)
        if not results:
            break
        yield results
        page += 1
```

2. Add to the dlt source:

```python
@dlt.source
def jira_source(jira_client: JiraClient):
    return [
        jira_issues(jira_client),
        jira_changelogs(jira_client),
        jira_my_new_endpoint(jira_client),  # ← add here
    ]
```

3. The table `raw_jira.my_new_endpoint` will be auto-created by dlt on first run.

---

## Write Dispositions

| Disposition | When to use | Effect |
|---|---|---|
| `replace` | Full re-sync every run | TRUNCATE + INSERT |
| `append` | Changelogs, additive data | INSERT only |
| `merge` | With `primary_key` — upsert | MERGE on key |

Issues use `merge` with `id` as primary_key. Changelogs use `append`.

## HTTP Client Patterns

The Jira HTTP client uses `tenacity` for retry:

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
def get_issues(self, project_key: str, start_at: int = 0) -> list[dict]:
    resp = self._session.get(
        f"{self.base_url}/rest/api/3/search",
        params={"jql": f"project={project_key}", "startAt": start_at, "maxResults": 50},
        timeout=(CONNECT_TIMEOUT, READ_TIMEOUT),
    )
    resp.raise_for_status()
    return resp.json()["issues"]
```

Timeouts from env:
- `JIRA_HTTP_CONNECT_TIMEOUT_SEC` (default: 5)
- `JIRA_HTTP_READ_TIMEOUT_SEC` (default: 60)

## Important: Clean Layer Does NOT Use dlt Watermarks

**TD-001 (known, not fixed):** The clean layer assets do NOT read `_dlt_load_id` to process only new records. Every clean layer run reads ALL of `raw_jira.*`.

This means:
- Clean layer runtime is O(total data), not O(new data)
- Re-running a clean asset is safe and idempotent (it overwrites clean_jira)
- Do NOT add watermark logic to clean assets without understanding the full impact on all downstream metrics

## Projects Configuration

Projects are configured in `config/projects.yaml` — dlt reads this to know which Jira projects to sync.
The `JIRA_PROJECTS` environment variable has NO effect on ingestion — it is unused in the actual code (a remnant).

```yaml
# config/projects.yaml
projects:
  - key: MYPROJ
    name: My Project
    jira_instance: default
    enabled: true
```

To add a new project: add it here, then run `raw_jira_data` asset.

## Resetting the dlt Watermark (CRITICAL — read before touching state)

**dlt stores watermark in TWO places simultaneously:**

| Location | Path | Precedence |
|---|---|---|
| PostgreSQL | `raw_jira._dlt_pipeline_state` (MAX version row) | Remote |
| Docker container file | `/home/dagster/.dlt/pipelines/<pipeline>/state.json` | Local |

On `pipeline.sync_destination()`, dlt takes **MAX(local, remote)**. If the container file has a newer watermark, it silently wins — making any DB-only reset invisible to the next run.

**Always reset BOTH with the script:**

```bash
python scripts/reset_dlt_watermark.py --all --date 2026-03-18
# --container defaults to "dagster"; pass --container "" to skip Docker reset
```

The script:
1. Reads latest state from `_dlt_pipeline_state`, patches `last_value + unique_hashes`, inserts new version row
2. `docker exec dagster cat/write` the local state file with the same reset

**Do NOT:**
- Edit only the host machine's `~/.dlt/` files — Dagster runs in Docker with its own home dir
- Insert DB state rows with `_dlt_load_id = '0'` — dlt ignores them (load_id must exist in `_dlt_loads`)
- Encode state blobs with gzip (`wbits=15+16`) — dlt decompresses with plain `zlib.decompress()` (zlib format only); gzip causes `incorrect header check`

**Symptoms of a stuck watermark:**
- `raw_jira._dlt_loads` shows recent loads with 0 rows loaded
- `raw_jira.issues` max `fields__updated` doesn't advance despite Jira having newer data
- `clean_jira.issues` `jira_updated_at` frozen at an old date

## Checking Raw Data Health

```python
# Quick check: how many issues in raw layer?
SELECT COUNT(*) FROM raw_jira.issues;

# Check last load timestamp
SELECT loaded_at, schema_name, table_name
FROM _dlt_loads
ORDER BY loaded_at DESC
LIMIT 10;

# Check for a specific project
SELECT COUNT(*) FROM raw_jira.issues
WHERE fields__project__key = 'MYPROJ';
```
